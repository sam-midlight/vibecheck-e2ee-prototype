# Video call design — v3 E2EE over LiveKit + SFrame

**Status:** planning (2026-04-16). No code written yet. Target: foundation-grade primitive suitable for verbatim lift into V2 and subsequent consuming apps.

## 1. Goals & non-goals

**Goals**
- End-to-end encrypted group video calls (2–N participants) scoped to existing rooms.
- Reuse the v3 per-device key infrastructure as the frame-key distribution primitive — the same sealed-box-to-device-x25519 pattern that wraps room keys wraps call keys.
- SFU never holds plaintext media. Supabase never holds plaintext keys.
- Revoking a device cascades into active calls: ghost devices cannot decrypt post-revoke frames.
- Foundation / reference-UX split preserved (§10).
- Retro aesthetic constraint: **QVGA (320×240 @ 15fps)** video, full-quality Opus audio, no simulcast.

**Non-goals (this iteration)**
- Recording / playback (requires plaintext access at some point; deferred).
- Push notifications for incoming calls (use realtime presence for now).
- PSTN / SIP bridging.
- Federation with external Matrix/XMPP video.
- Large-scale calls (>20 participants). Design should not preclude, but initial targets are 2–8.

## 2. Architectural layering

```
┌──────────────────────────────────────────────────────────────┐
│ Reference UX:  call page, video tiles, mute/camera, CRT FX   │
├──────────────────────────────────────────────────────────────┤
│ Foundation:    e2ee-core/call.ts    (key gen, wrap, rotate)  │
│                bootstrap.ts         (LiveKit key-provider)   │
│                supabase/queries.ts  (call CRUD + RPC)        │
├──────────────────────────────────────────────────────────────┤
│ LiveKit client SDK:  SFrame framing, track pub/sub, transport│
├──────────────────────────────────────────────────────────────┤
│ Supabase:      calls/call_members/call_key_envelopes + RPCs  │
│ LiveKit SFU:   zero-plaintext media forwarding               │
│ Edge function: LiveKit JWT minting with device verification  │
└──────────────────────────────────────────────────────────────┘
```

Same split as the rest of the codebase. `e2ee-core/call.ts` is pure crypto, no React, no Supabase. LiveKit integration lives in a thin adapter layer alongside `bootstrap.ts`.

## 3. Key hierarchy

| Key | Scope | Source | Lifetime |
|---|---|---|---|
| UMK (existing) | per user | already provisioned | rotated only on explicit rotation |
| DeviceKeyBundle (existing) | per device | already provisioned | per-device lifetime |
| **CallKey** (new) | `(call_id, generation)` | freshly generated by rotator | ephemeral, per-generation |

`CallKey` is a 32-byte symmetric AEAD key. It is **not** derived from any existing room key — a call is its own scope, and its membership is a subset of the room at any instant. Deriving from the room key would couple call forward-secrecy to room rotation, which is the wrong lifecycle.

Frame keys passed to LiveKit's SFrame layer are derived *by the SDK* from the CallKey via its built-in HKDF. We hand LiveKit a single 32-byte key and a `keyIndex` (= generation); the SDK handles per-track/per-frame derivation internally.

## 4. Schema — migration 0023

```sql
create table calls (
  id uuid primary key default gen_random_uuid(),
  room_id uuid not null references rooms(id) on delete cascade,
  initiator_user_id uuid not null references auth.users(id),
  initiator_device_id uuid not null references devices(id),
  started_at timestamptz not null default now(),
  ended_at timestamptz,                    -- null = active
  current_generation int not null default 1
);

create table call_members (
  call_id uuid not null references calls(id) on delete cascade,
  device_id uuid not null references devices(id),
  user_id uuid not null references auth.users(id),
  joined_at timestamptz not null default now(),
  left_at timestamptz,
  last_seen_at timestamptz not null default now(),  -- heartbeat (§6.5)
  primary key (call_id, device_id)
);

create table call_key_envelopes (
  call_id uuid not null references calls(id) on delete cascade,
  generation int not null,
  target_device_id uuid not null references devices(id),
  sender_device_id uuid not null references devices(id),
  ciphertext bytea not null,   -- sealed-box(CallKey → target device_x25519_pub)
  signature bytea not null,    -- ed25519 sign over (call_id || generation || target_device_id || ciphertext)
  created_at timestamptz not null default now(),
  primary key (call_id, generation, target_device_id)
);
```

**RLS**
- `calls.select` — room members only (via `is_room_member_at` helper)
- `calls.insert/update` — RPC-only (no direct client writes)
- `call_members.select` — room members
- `call_members.insert/update` — RPC-only
- `call_key_envelopes.select` — `target_device_id = current device of auth.uid()`
- `call_key_envelopes.insert` — RPC-only with server-side signature verification

The pattern mirrors `room_members` + `room_invites` — no direct-client writes to anything security-critical, all changes flow through signed-and-verified RPCs.

## 5. RPC surface

- `start_call(p_call_id uuid, p_room_id uuid, p_envelopes jsonb)` — creates `calls` row with client-supplied `call_id` (UUIDv7), inserts `call_members` for each target in envelopes, inserts envelopes atomically. Verifies caller is current-gen room member. Rejects on `call_id` collision.
- `rotate_call_key(p_call_id uuid, p_new_generation int, p_envelopes jsonb)` — bumps `calls.current_generation`, replaces envelopes for the new generation. Verifies caller is an active call member. **Requires `p_new_generation = current_generation + 1`** — this is the primary arbiter for the race case in §6.3.
- `join_call(p_call_id uuid)` — inserts `call_members` row for caller's device; returns current generation + caller's envelope if one exists yet, or signals "awaiting rotation."
- `leave_call(p_call_id uuid)` — sets `call_members.left_at`. Does not itself bump generation — the elected rotator does that (§6.4).
- `heartbeat_call(p_call_id uuid)` — updates `call_members.last_seen_at`. Called every 10s by active clients; stale rows (>30s) are treated as disconnected by the reconnection-grace logic (§6.5).
- `end_call(p_call_id uuid)` — initiator or last-remaining member sets `calls.ended_at`. Cascade purges envelopes.

All RPCs `SECURITY DEFINER` with explicit `auth.uid()` checks, matching `kick_and_rotate` style.

## 6. End-to-end flow

### 6.1 Alice starts a call in room R

1. **Enumerate targets.** Client fetches all `devices` rows for users in `room_members` at `current_generation(R)`. Filters out revoked devices (cert revocation check, existing primitive).
2. **Generate CallKey.** 32 random bytes from libsodium.
3. **Pre-generate `call_id`.** UUIDv7 client-side. This lets us sign `(call_id || generation || target || ciphertext)` deterministically before the server knows about the call.
4. **Wrap per device.** For each target device: `sealed_box(CallKey, target.device_x25519_pub)` → ciphertext. Sign `(call_id || 1 || target_device_id || ciphertext)` with Alice's `device_ed25519_priv`.
5. **`start_call` RPC.** Atomic: inserts `calls`, `call_members` rows for all targets, and `call_key_envelopes` rows. Rolls back if any signature fails verification or `call_id` collides.
6. **Get LiveKit JWT.** Alice hits edge function `/livekit-token` with `{call_id, device_id}`. Edge function verifies device membership and mints short-lived JWT.
7. **Connect to LiveKit.** `new Room({ e2ee: {...} })`, `setKey(CallKey, keyIndex=1)`, `setE2EEEnabled(true)`, `connect(url, jwt)`.
8. **Publish QVGA+Opus.** Capture constraints force 320×240@15fps, simulcast=off, videoEncoding.maxBitrate=200kbps.
9. **Realtime broadcast.** Supabase realtime channel `room:{R}` receives `{type: 'call_started', call_id}`.

### 6.2 Bob receives and joins

1. Bob's client is subscribed to `room:{R}` realtime. Receives `call_started`.
2. **Fetch envelope.** `call_key_envelopes` where `(call_id, generation=1, target_device_id=bob_device)`.
3. **Verify signature chain.** `envelope.signature` verified against `envelope.sender_device_id`'s ed25519 pub; sender's device cert verified against sender's UMK; TOFU check (existing `observeContact` primitive).
4. **Unwrap.** `sealed_box_open(envelope.ciphertext, bob_device_x25519_{pub,priv})` → CallKey.
5. **`join_call` RPC.** Inserts Bob's `call_members` row.
6. **LiveKit JWT, connect, setKey, publish.** Same as §6.1 steps 6–8.

### 6.3 Carol joins mid-call — rotator election

1. Carol's client: `join_call(call_id)` — RPC returns `{current_generation: G, envelope: null}` (no envelope for Carol yet).
2. Realtime broadcast `{type: 'member_joined', call_id, device_id: carol}` to existing participants.
3. **Rotator election (optimistic, DB-arbitrated).** Every currently-connected member independently checks whether they are the lowest-ordered active device by `(joined_at ASC, device_id ASC)`. If yes, they proceed to rotate. If no, they wait for the rotation broadcast.
4. The elected rotator generates `CallKey'`, wraps to current members + Carol, and calls `rotate_call_key(call_id, G+1, new_envelopes[])`.
5. **Race resolution:** if two clients both believed they were the rotator and both submit `rotate_call_key` with `p_new_generation = G+1`, the RPC's uniqueness constraint (`p_new_generation = current_generation + 1`) causes the second to fail. The loser observes via the `key_rotated` broadcast, reads the new generation, and gives up the attempt. **No leader lease, no coordination.** This is the Matrix/CRDT pattern: let the DB be the arbiter.
6. If the elected rotator is silent for 2 seconds after `member_joined`, the next-in-order device attempts. Same race semantics apply — worst case is one spurious failed RPC, no correctness impact.
7. All clients receive `{type: 'key_rotated', call_id, generation: G+1}` via realtime, fetch their new envelope, call `keyProvider.setKey(CallKey', G+1)`.
8. LiveKit SDK handles the cutover: old key stays valid for a short grace window for in-flight frames, new key active for new frames.

**Property:** Carol cannot decrypt frames sent before she joined, because those were encrypted under `CallKey` and her envelope is only for `CallKey'`.

### 6.4 Leave + revoke cascade

**Graceful leave:**
1. Dave's client: `leave_call(call_id)` — marks `left_at`.
2. Realtime broadcast `{type: 'member_left', call_id, device_id: dave}`.
3. Rotator election (§6.3 rule, with Dave excluded from the ordered set) triggers `rotate_call_key` with Dave **excluded** from new envelopes.
4. From the next frame onward, Dave cannot decrypt even if he re-opens the LiveKit connection.

**Device revocation (security-critical):**
1. User (from primary device) issues `revoke_device(device_id)` — existing v3 flow, publishes UMK-signed revocation cert.
2. **New cascade step:** for each active `call` where that `device_id` is a `call_member` with `left_at IS NULL`, trigger the rotator election to rotate immediately.
3. Revoked device's LiveKit JWT is already short-lived; on next renewal (§7.3), edge function checks cert revocation and refuses.
4. Even before JWT refresh, the revoked device's `CallKey` is stale → all frames post-rotation are undecryptable.
5. **`revokeDevice` in bootstrap.ts must include this cascade**, analogous to how `rotateUserMasterKey` calls `rotateAllRoomsIAdmin`.

### 6.5 Reconnection grace — the flap policy

Mobile connections drop constantly — WiFi↔cellular handoff, elevators, backgrounding. Bumping the CallKey on every websocket blip would cause constant A/V dropouts while keys re-sync. **Decision: 30-second grace window.**

1. Active clients heartbeat via `heartbeat_call(call_id)` every 10 seconds. Server updates `call_members.last_seen_at`.
2. When a client disconnects (realtime presence drop or heartbeat stale), other clients **do not immediately** trigger rotation. They start a 30-second timer keyed on that `device_id`.
3. If the missing device's heartbeat resumes within 30 seconds, timer is cancelled, no rotation, seamless continuation.
4. If 30 seconds elapse without heartbeat, the elected rotator treats the device as having left and triggers `rotate_call_key` with it excluded. This mirrors the graceful-leave path.
5. **Security note:** during the 30-second grace, a maliciously offlined-then-re-onlined device retains the CallKey. This is a deliberate tradeoff — the alternative is UX hostility on normal network flaps. 30s is short enough that the attack window is tight; explicit `revoke_device` immediately forces a rotation regardless of grace state.

## 7. LiveKit integration

**Why LiveKit vs building our own SFU:** Signal spent ~2 years building theirs. Element Call adopted LiveKit rather than building. The `ExternalE2EEKeyProvider` primitive is designed precisely for our bring-your-own-key model. Building our own SFU is a multi-person-year project that buys us nothing a consuming app would benefit from.

**LiveKit deployment:** start with LiveKit Cloud for prototype (free tier, 60-min call cap, sufficient for dev). Self-host path is `livekit-server` + Redis, documented in LiveKit repo. Consuming apps choose their own deployment.

### 7.1 Edge function — `/livekit-token`

```
POST /functions/v1/livekit-token
Auth: Supabase user JWT
Body: { call_id: uuid, device_id: uuid }

Server logic:
  1. Verify user's session → user_id
  2. Verify device_id ∈ devices where user_id matches AND revocation IS NULL
  3. Verify (call_id, device_id) ∈ call_members with left_at IS NULL
  4. Mint LiveKit JWT:
       sub:         `${user_id}:${device_id}`
       room:        `call:${call_id}`
       ttl:         5 minutes
       canPublish:  true
       canSubscribe: true
     Sign with LIVEKIT_API_SECRET (edge function env var)
  5. Return { jwt, url: LIVEKIT_WS_URL, expiresAt }
```

Short TTL (5 min) is the enforcement mechanism for cross-call revocation. A freshly revoked device loses SFU access within one refresh cycle (≤5min) even if its current connection is live.

### 7.2 Client setup

```ts
const keyProvider = new ExternalE2EEKeyProvider();
const room = new Room({
  e2ee: {
    keyProvider,
    worker: new Worker(new URL('livekit-client/e2ee-worker', import.meta.url))
  },
  videoCaptureDefaults: {
    resolution: { width: 320, height: 240, frameRate: 15 }
  },
  publishDefaults: {
    videoEncoding: { maxBitrate: 200_000, maxFramerate: 15 },
    simulcast: false,
    audioPreset: AudioPresets.speech   // 32 kbps Opus
  }
});
await keyProvider.setKey(callKey, /*keyIndex=*/ generation);
await room.setE2EEEnabled(true);
await room.connect(livekitUrl, jwt);
```

On rotation: `await keyProvider.setKey(newCallKey, newGeneration)`. LiveKit handles the handover.

### 7.3 Silent token renewal — critical for long calls

**The problem:** a 45-minute call with 5-minute JWTs means **9 token renewals**. If renewal isn't automatic, users randomly drop out. The short TTL is a security feature; it is the consuming app's job to make it invisible.

**Adapter responsibility.** `livekit-adapter.ts` owns the renewal loop. The call-screen UI must never need to think about tokens.

**Protocol:**
1. On `connect()`, adapter records `expiresAt` from the token response.
2. Adapter schedules a renewal timer to fire at `expiresAt - 60s` (i.e. minute 4 of a 5-min token). One-minute buffer absorbs clock skew + network latency.
3. On fire: POST `/livekit-token` with `{call_id, device_id}`, get new JWT.
4. Pass new JWT to LiveKit SDK via `room.localParticipant.setMetadata()` or the SDK's re-auth hook (confirm exact API during implementation — LiveKit has evolved this; check SDK version's docs).
5. Schedule next renewal on the new `expiresAt`.
6. Clear the renewal timer in `room.disconnect()` / on unmount.

**Failure modes & handling:**
- **Renewal endpoint 401/403 (device revoked mid-call):** adapter disconnects the room, emits a `call_ended_revoked` event. UI shows "your device was revoked" banner. This is the intended revocation cascade in action.
- **Transient network failure at renewal:** retry with exponential backoff — 1s, 2s, 4s — until existing token expires. If expired before success, disconnect and let the user manually rejoin.
- **Tab backgrounded / timer throttled:** modern browsers throttle `setTimeout` in background tabs. Use a `visibilitychange` listener to force an immediate renewal check on tab foreground; otherwise backgrounded calls could miss their renewal window and silently drop.
- **Clock skew >60s:** the 60s buffer is the absorbed margin. On observed token rejection despite "valid" local time, immediately renew and adjust the buffer for this session.

**Status-check probe (§12 phase 6 addition):** include a "token renews silently across a 5-min boundary" test in the status page's green-dot regression harness. Without an automated probe, the bug "call drops at exactly 5:00" would be a silent regression.

## 8. QVGA / retro aesthetic policy

- `videoCaptureDefaults.resolution = { 320, 240, 15fps }` — hardcoded in foundation's call bootstrap. Consuming apps can override per-call but the default is the retro mode.
- `simulcast: false` — single layer, no multi-resolution. At QVGA there's nothing to downscale to, and it eliminates a key-management wrinkle (each simulcast layer gets its own frame key).
- `maxBitrate: 200_000` — hard cap. Typical QVGA@15fps runs ~100–150 kbps; cap gives headroom for motion spikes.
- Audio: `AudioPresets.speech` (32 kbps Opus). **Do not downgrade audio to match video aesthetic** — crunchy audio reads as "broken," not "retro."
- Optional CRT/scanline/pixel-doubling effects: composable as a reference-UX layer via `canvas.captureStream()` → feed the canvas MediaStreamTrack as the LiveKit publish source. Purely visual; no crypto impact.
- **Honesty:** QVGA enforcement is client-side convention, not cryptographically enforced. A modified client could publish HD. Same caveat as "client can screen-record their own video": not a crypto property, just an app norm.

## 9. Security properties

| Property | Mechanism |
|---|---|
| SFU cannot read media | SFrame E2EE on all published frames; LiveKit SFU sees only opaque ciphertext |
| Signaling server cannot read keys | CallKey sealed-box-wrapped to device X25519 pub; Supabase only ever sees ciphertext |
| Call-invite spoofing blocked | Envelopes signed by `sender_device_ed25519`; signature verified against device cert → UMK chain |
| Ghost device cannot decrypt post-revoke | Device revocation triggers `rotate_call_key`; ghost has no envelope for new generation |
| Forward secrecy on join | New member can't decrypt pre-join frames (different key generation) |
| Backward secrecy on leave | Departed member can't decrypt post-leave frames (different key generation) |
| SFU cannot forge participants | LiveKit JWT sub-claim is `{user_id}:{device_id}`; edge function only mints for authenticated+unrevoked+call-member devices |
| Cross-call access rapidly revoked | 5-min JWT TTL + silent renewal that re-checks revocation |
| Hostile SFU cannot inject frames | SFrame AEAD auth tag binds to CallKey; injected frames fail verification |
| TOFU applies to call auth | Envelope signature verification uses the same `observeContact` flow as messages → any UMK drift surfaces in the existing KeyChangeBanner UX |

### Residual risks (acknowledged, deferred)
- **Traffic analysis.** SFU sees who's in which call and when. Unavoidable without onion-routed media (out of scope).
- **Malicious client recording.** Any participant can record their own decrypted stream. Same as Signal / Matrix / every other E2EE video system. Crypto doesn't fix this.
- **LiveKit client SDK integrity.** We trust the SDK not to leak CallKey. Mitigation: SRI-pin the bundle, review updates. In V2, consider bundling a vendored copy.
- **30s reconnection grace.** Deliberate UX/security tradeoff (§6.5). Explicit revocation bypasses the grace.
- **Metadata in `calls.started_at` etc.** Who called whom when is stored plaintext in the DB. Matches the existing `room_members.joined_at` precedent — this is consistent with the prototype's threat model, not a regression.

## 11. Open questions

1. ~~**Rotator election robustness.**~~ **Resolved:** optimistic election + `rotate_call_key`'s `p_new_generation = current_generation + 1` uniqueness constraint arbitrates races (§6.3). No leader lease. Matrix/CRDT pattern.

2. **Incoming-call UX.** Realtime presence is fine for app-open callees. For app-closed → push notifications. Deferred to a later iteration; noted in roadmap.

3. ~~**Reconnection handling.**~~ **Resolved:** 30-second heartbeat-grace window (§6.5). Bump only on true leave or heartbeat staleness past grace. Explicit `revoke_device` still forces an immediate rotation.

4. **Call duration / recording retention.** No recording this iteration. For call-list UI, decide: keep `calls` rows forever (searchable history) or purge after N days? Leaning keep, since they're just metadata.

5. **LiveKit deployment choice.** Cloud vs self-host vs "consuming app picks." Prototype: Cloud. V2: self-host (`livekit-server` in same infra as Supabase). Document both paths.

6. **Mobile Safari WebRTC quirks.** Safari requires user-gesture-initiated `getUserMedia` and has permission-state quirks. Validate during phase 4 implementation; may require UX shaping.

7. **LiveKit SDK token-refresh API.** §7.3 notes the exact SDK surface (`setMetadata` vs a dedicated re-auth hook) depends on SDK version. Pin the version during phase 3 and lock the API choice in `livekit-adapter.ts`.

## 12. Phased implementation

1. **Migration 0023** — schema, RLS, RPCs. SQL-testable in isolation. Lands alone.
2. **Edge function** — LiveKit JWT minting. Test with `curl` + manual LiveKit SDK.
3. **Foundation layer** — `call.ts` + `livekit-adapter.ts` (including renewal loop) + queries + bootstrap helpers. Unit-testable without UI.
4. **Minimal 2-participant call** — barebones UI, proves the stack E2E. Two browser windows, one room, one call.
5. **Mid-call join** — rotator election + generation bump.
6. **Leave + revoke cascade + renewal probe** — the security-critical flows. Must include status-check probes:
   - Revoked device cannot decrypt post-revoke test frame.
   - Silent token renewal across a 5-min boundary without user-visible drop.
7. **Reconnection-grace validation** — simulate 10s and 40s disconnects; confirm no rotation on the former, rotation on the latter.
8. **QVGA polish + retro effects** — canvas pipeline, visual layer. Last, because it's pure UX.
9. **Port-to-v2.md update** — migration 0023 entry, new foundation surfaces, revocation-cascade invariant, renewal-loop invariant.

## 13. What this does not change

- v3 per-device identity model: untouched.
- Room key machinery: untouched. Calls are an orthogonal scope.
- `kick_and_rotate` RPC: untouched. `rotate_call_key` is structurally analogous but operates on `call_members`, not `room_members`.
- Recovery phrase / backup-key flow: untouched. Calls are ephemeral; no call state is backed up.
- Existing TOFU (`observeContact`): reused for envelope signature verification. No new trust primitive.

The entire design is additive. No existing invariant changes.

## 14. Post-launch deltas (2026-04-16, same-day)

Things that differ from the design as originally drafted — captured here so future readers don't chase "why didn't we do X." See `memory/project_video_calls_2026_04_16.md` for the session arc.

- **Edge function `verify_jwt` is `false`, not `true`.** Supabase Edge gateway is HS256-only; projects with ES256-signed auth tokens (the newer default) get rejected upstream with `UNAUTHORIZED_UNSUPPORTED_TOKEN_ALGORITHM`. Our function verifies the session *inside the handler* via `supabase.auth.getUser()`, which is algorithm-agnostic. Functional equivalence to `verify_jwt: true` with broader project compatibility.
- **Env var is `LIVEKIT_URL`, not `LIVEKIT_WS_URL`.** LiveKit Cloud's dashboard ships `LIVEKIT_URL`; the function accepts either for backward-compat.
- **Token fetcher uses plain `fetch` with explicit `apikey` + `Authorization` headers**, not `supabase.functions.invoke`. The invoke helper omits `apikey` on some SDK versions and gets gateway-rejected. Lesson for V2: for any future edge function relying on user auth, use the same pattern.
- **E2EE worker is a static asset at `public/livekit-e2ee-worker.mjs`.** Turbopack can't resolve bare module specifiers for `new Worker(new URL('livekit-client/e2ee-worker', import.meta.url))` and throws cryptic `e.indexOf is not a function`. A `postinstall` + `prebuild` script copies the SDK's worker into `public/`.
- **Encryption-error tolerance window.** The adapter now treats a burst of `RoomEvent.EncryptionError` (up to 10 in 10s) as transient — nudging the key provider at 3 errors, only flipping to 'failed' at 10. A single `InvalidKey: Decryption failed` during rotation is normal; treating it as terminal would kill otherwise-healthy calls.
- **`room.isE2EEEnabled` cannot be asserted synchronously after `setE2EEEnabled(true)`.** The flag flips asynchronously via the `ParticipantEncryptionStatusChanged(local, enabled=true)` event, which only fires once the first encrypted track publishes. The adapter now listens for the event and `publishLocalMedia` awaits it with a 15s timeout.
- **Zombie-call seamless takeover.** If `fetchAndUnwrapCallKey` returns null on join AND no other active members are present (per `filterActiveCallMembers` + 30s heartbeat grace), the joiner self-rotates to take ownership. If others exist but fail to rotate us in within 7s, a watchdog self-rotates as fallback. This wasn't in the original §6 flow; added after testing.
- **Tab-close / unmount cleanup.** `pagehide` fires a `fetch({keepalive: true})` leave_call; component unmount also calls `leave_call` RPC. Cuts the zombie-member window from 30s (heartbeat grace) to ~0.
- **Migration 0024 partial unique index** prevents two simultaneous `start_call` RPCs from producing parallel active call rows. Not in the original schema; added as reliability polish.
- **Status probe "Browser supports E2EE insertable streams"** added as probe #17 — fails loudly if the browser lacks the API entirely, since LiveKit would silently fall back to plaintext SRTP in that case.
